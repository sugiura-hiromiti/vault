# TRPL 第2章: 数当てゲームのプログラミング - 教材

## 議論のポイント

### 1. プロジェクト構造とCargoワークフロー
- **Cargoでのプロジェクト設定**: `cargo new`が完全なプロジェクト構造を作成する方法
- **開発ワークフロー**: `cargo run`、`cargo build`、`cargo check`のサイクル
- **依存関係管理**: Cargo.tomlとCargo.lockが連携して再現可能なビルドを実現する仕組み

### 2. 入出力と標準ライブラリ
- **preludeの概念**: Rustがデフォルトでスコープに持ち込むものと明示的なインポート
- **標準ライブラリの構成**: `std::io`とモジュール構造の理解
- **ユーザー入力の処理**: `stdin().read_line()`パターンと`Result`を返す理由

### 3. 変数と可変性
- **デフォルトで不変**: 不変性を通じたメモリ安全性へのRustのアプローチ
- **シャドウイング vs 可変性**: `let`と`let mut`の使い分けと変数のシャドウイング
- **型推論**: Rustが型を推論する方法と明示的な型注釈が必要な場合

### 4. エラーハンドリングの哲学
- **Result型**: `Ok`と`Err`バリアントの理解
- **明示的なエラーハンドリング**: Rustが潜在的な失敗の処理を強制する理由
- **`expect()` vs `match`**: エラーハンドリングの異なるアプローチとそのトレードオフ

### 5. パターンマッチングと制御フロー
- **match式**: パターンマッチングの動作と網羅性の理由
- **Ordering列挙型**: 列挙型を使用した状態と比較結果の表現
- **ループ制御**: プログラムフローのための`loop`、`break`、`continue`

### 6. 型システムと変換
- **文字列型と数値型**: 異なる型を直接比較できない理由
- **パースと変換**: `parse()`メソッドと型注釈
- **型安全性**: Rustがコンパイル時に一般的なプログラミングエラーを防ぐ方法

### 7. 外部依存関係
- **クレートエコシステム**: バイナリクレートとライブラリクレートの理解
- **セマンティックバージョニング**: Cargo.tomlでのバージョン指定子の動作
- **ドキュメント**: `cargo doc --open`を使用したクレートAPIの探索

## 解答（クイズ）

### 基本概念
1. b) Cargo.tomlとsrc/main.rsを含む完全なプロジェクト
2. 偽（変数はデフォルトで不変）
3. `std::io`
4. c) Result<usize, Error>（読み取ったバイト数を返す）
5. 最初は不変変数を作成し、2番目は後で変更可能な可変変数を作成します。

### 中級概念
6. 型の不一致：`secret_number`は整数、`guess`は文字列。直接比較できません。
7. b) 値を無視するキャッチオールパターン
8. `trim()`はユーザーがEnterを押したときに追加される改行文字を含む空白文字を削除します。
9. 
   ```rust
   match guess.cmp(&secret_number) {
       Ordering::Less => println!("小さすぎます！"),
       Ordering::Greater => println!("大きすぎます！"),
       Ordering::Equal => {
           println!("正解です！");
           break;
       }
   }
   ```

### 高度な理解
10. 異なるエラーケースを異なって処理したい場合、またはpanicではなく実行を継続したい場合に`match`を使用します。`expect()`はより簡単ですが柔軟性に欠けます。

11. `cargo update`は依存関係を最新の互換バージョンに更新し、Cargo.lockを更新します。`cargo build`は依存関係が変更されていない限り、既存のCargo.lockバージョンを使用します。

12. `parse()`は異なる数値型を返すことができるため、Rustはパースしたい具体的な型を知る必要があります。
